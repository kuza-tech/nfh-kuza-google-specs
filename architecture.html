1. Project Overview

NFH-Kuza-Google is a rural service delivery marketplace that connects rural households with local service providers through WhatsApp and a lightweight web application. The platform is built on the Beckn open network protocol, enabling decentralized discovery and fulfillment of services across four domains: livelihood services, household services, market produce trading, and agricultural advisory.

Design Principle

100% of farmer interaction occurs through WhatsApp — no app installation required. Providers use a lightweight Ionic Angular web app opened via WhatsApp button links.

Service Domains

  • Livelihood Services — agricultural inputs, equipment rental, veterinary services, soil testing, etc.
  • Household Services — plumbing, electrical, cleaning, construction, etc.
  • Market Produce — connecting farmers with aggregators and buyers (reversed economics: buyer pays farmer)
  • Advisory — agricultural content and guidance delivered as media through WhatsApp (no order lifecycle)

3. Architecture Layers

3.1 Layer 1: Frontend Interfaces

WhatsApp (Farmer-Facing)

WhatsApp is the sole farmer-facing interface. 100% of farmer interactions — service discovery, provider selection, order confirmation, payment declaration, and feedback — occur through WhatsApp messages.

  • Interactive buttons for small binary/ternary choices (Accept/Reject, Yes/No, payment method selection)
  • Numbered text lists for longer option sets (provider lists, service categories, sub-categories)
  • Free text input for location, quantity, and other open-ended fields
  • Connected via WhatsApp Cloud API

Kuza Web App (Provider-Facing)

A lightweight Ionic Angular web application opened from WhatsApp button links — no native app installation required. Providers use it to:

  • Manage their service catalog (add/edit services, set pricing)
  • View and accept/reject incoming orders
  • Update order status (in-progress, completed)
  • Report payment received
  • Manage their profile and service areas

3.2 Layer 2: WhatsApp Handler

Design Constraint

The existing webhook.php remains UNTOUCHED. Only routing logic is added to branch marketplace conversations to the new beckn_marketplace_handler.php based on trigger words or message IDs.

  • File: beckn_marketplace_handler.php (new PHP file on Kuza Platform)
  • State machine: manages conversational flow for the 4 marketplace domains (Livelihood, Household, Market, Advisory)
  • Translates WhatsApp interactions into BAP API calls
  • Receives async callbacks from BAP and formats WhatsApp response messages
  • Own tables: conversation state, session tracking — separate from existing poultry chatbot tables (chatbot_nodes, chatbot_transitions)
  • Provider notifications are triggered via a messaging queue/cron job, NOT through webhook.php

3.3 Layer 3: BAP (Beckn Application Platform)

The BAP is the farmer-side Beckn participant. It acts as the bridge between the WhatsApp Handler and the Beckn network.

  • Receives translated requests from the WhatsApp Handler
  • Constructs Beckn v2 protocol messages: discover, select, init, confirm, status, cancel, rate
  • Sends protocol messages through the ONIX-BAP adapter
  • Receives async callbacks (on_discover, on_select, on_init, on_confirm, on_status, on_cancel, on_rate)
  • Pushes results back to WhatsApp Handler for farmer delivery
  • Hosted by Kuza on GCP

3.4 Layer 4: BPP (Beckn Provider Platform)

The BPP is the provider-side Beckn participant. It manages the provider catalog, order lifecycle, and serves the provider web interface.

  • Receives Beckn protocol requests from the network (via ONIX-BPP adapter)
  • Connects to Kuza Platform to fetch provider catalogs, service areas, and pricing
  • Manages order lifecycle on the provider side (create, accept, update status, complete)
  • Sends async Beckn responses back through ONIX-BPP adapter
  • Serves the Kuza Web App (provider interface) for catalog and order management
  • Hosted by Kuza on GCP

3.5 Layer 5: Beckn Network Infrastructure

All Beckn network components are external to Kuza infrastructure, provided by Beckn Fabric.

ONIX Adapters

  • Plugin-based Go servers running the fidedocker/onix-adapter Docker image
  • Two instances: ONIX-BAP (farmer side) and ONIX-BPP (provider side)
  • Handle message signing, signature validation, schema validation, and network routing
  • Hosted on GCP as Docker containers

DeDi Registry

  • Endpoint: fabric.nfh.global/registry/dedi
  • Stores participant identity, public keys, and callback URIs
  • Used for signature validation and dynamic routing between BAP and BPP

Catalog Service

  • Endpoint: fabric.nfh.global/beckn/catalog
  • BPP publishes provider catalogs here
  • Queried by Discovery Service during discover requests

Discovery Service

  • Handles discover requests from BAP
  • Queries Catalog Service for matching providers
  • Returns results to BAP via on_discover callback

3.7 Layer 7: Google Cloud Services

Google contributes to the platform across three distinct service tiers: cloud infrastructure for hosting, content APIs for serving curated domain content, and AI interfaces for intelligent processing. Each is consumed at a different point in the pipeline.

3.7.1 Google Cloud Platform (GCP) — Hosting Infrastructure

The entire Kuza-built stack is deployed on GCP. This is the foundational layer on which everything Kuza builds runs.

  • Compute: hosts the WhatsApp Handler, BAP application, BPP application, Kuza Web App, ONIX adapters (Docker)
  • Database: hosts MySQL 8 (the Kuza Platform data store)
  • Networking: public endpoints, TLS termination, load balancing for BAP / BPP callbacks
  • Co-location benefit: Kuza application code and Google Content APIs / AI interfaces are co-located on the same cloud, minimising latency for in-flow API calls

3.7.2 Google Content APIs — Curated Domain Content

Google publishes a set of curated content APIs per domain (agri, health, market, finance, etc.) that serve informational / educational content to farmers via the chatbot. These are separate from Beckn transactional catalog data — they don't list providers or services for sale, they deliver knowledge.

  • Primary consumer: Advisory flow — the BPP calls Google Content APIs (and Google AI) during on_discover processing to source advisory material. Content may include curated articles, videos, audio, infographics, or AI-generated summaries.
  • Secondary consumers: Livelihood / Household / Market flows — these can enrich a transaction with supplementary content (e.g., "Best practices for vaccinating broilers" attached to a Vaccination order; "Egg storage tips" attached to a Market produce sale). Used optionally to add value beyond the transaction.
  • Domain coverage: agri, health, market, finance, education — matched to Kuza's flow categories. Each domain has its own taxonomy of categories and sub-categories that align with the master data taxonomy in Kuza Platform.
  • Delivery format: APIs return content payloads (text, media URLs, structured guides) that the BPP wraps into the Beckn on_discover callback for Advisory, or surfaces as supplementary messages in the chatbot for other flows.
  • Called from: BPP application logic (during request processing) and optionally from the WhatsApp Handler for inline content.

3.7.3 Google AI Interfaces — Intelligent Processing

Distinct from content APIs, the AI interfaces provide computational services that process input and return derived results.

  • Location normalization (all flows): Farmer enters free-text location (village / ward / county) → PHP handler sends text to Google AI → AI maps to structured County + Sub-County → PHP handler sends confirmation message to farmer on WhatsApp → farmer confirms or corrects.
  • Advisory content generation (Advisory flow): When curated content from the Content APIs needs personalisation, summarisation, or translation, the BPP combines Content API output with farmer context and passes it through the Google AI interface to produce the final response delivered to the farmer.
  • Future: intent extraction from free-text farmer queries, language translation (English ↔ Swahili), sentiment analysis on feedback.
  • Called from: PHP handler (location normalization) and BPP (advisory generation, future intent extraction).
Content APIs vs AI Interfaces — how they relate: Content APIs serve what we know (curated knowledge bases per domain). AI Interfaces serve what we infer (normalising, generating, translating). The Advisory flow typically uses both: Content API supplies the source material, AI Interface tailors it for the specific farmer.

    Farmer confirms [Yes]
    --> BAP Client --> ONIX-BAP --> [Beckn Network / Discovery Service] --> ONIX-BPP --> BPP
      --> Kuza Platform (query providers by location + category + subCategory)
    <-- BPP --> ONIX-BPP --> [Beckn Network] --> ONIX-BAP --> BAP Client
  <-- beckn_marketplace_handler.php --> WhatsApp (list of providers with prices)

(continues through select / init / confirm / status / update / cancel / rate / support —

Order confirmed in Beckn
  --> BPP --> Kuza Platform (create order record)
  --> Messaging Queue / Cron --> WhatsApp Cloud API --> Provider WhatsApp
    Provider taps [Accept] button in WhatsApp
      --> webhook.php --> provider routing --> BPP --> Beckn update
    Provider taps [View Details] link
      --> Opens Kuza Web App in browser --> reads from Kuza Platform

  --> Marks service complete, enters payment details, notes

  --> Order marked FARMER_CONFIRMED --> CLOSED

Farmer (WhatsApp): Selects Advisory --> Domain (Agri/Health/Market/Finance) --> Category --> Sub-category
  --> webhook.php --> beckn_marketplace_handler.php
    --> BAP Client --> ONIX-BAP --> [Beckn Network] --> ONIX-BPP --> BPP

      ┌── BPP processing ──────────────────────────────────────────┐
      │ 1. BPP calls Google Content API for {domain, category,     │
      │    sub-category} --> returns curated content payload       │
      │    (articles, video URLs, guides)                          │
      │ 2. (Optional) BPP calls Google AI Interface to             │
      │    personalise / summarise / translate using farmer        │
      │    context (location, enterprise, language)                │
      │ 3. BPP composes the final advisory response                │
      └────────────────────────────────────────────────────────────┘

    <-- BPP --> ONIX-BPP --> [Beckn Network] --> ONIX-BAP --> BAP Client
  <-- beckn_marketplace_handler.php --> WhatsApp (curated/AI-tailored advisory)

Google Content APIs supply the source material; Google AI tailors it.
Advisory is free, no order lifecycle. Content delivered immediately via on_discover.

Flow E: Market Produce (Farmer Sells)

Same flow as A but with reversed economics:
- Farmer has produce to sell (quantity + type)
- discover returns aggregators/buyers with buying prices
- Aggregator collects produce from farmer
- Aggregator pays farmer (not farmer pays provider)
- Both parties confirm and provide feedback

(continues through the same full Beckn lifecycle —

4.6 Complete Beckn Endpoint Mapping

This section maps every user-facing step in each flow to the corresponding Beckn v2 protocol endpoint, covering the full order lifecycle — discovery, selection, initialization, confirmation, fulfilment status, in-flight updates, cancellation, rating/feedback, and support.

Beckn endpoints reference:

  • discover / on_discover — find services/providers in the network
  • select / on_select — request a priced quote for a specific item
  • init / on_init — submit billing & fulfilment details, receive draft order
  • confirm / on_confirm — create the binding order
  • status / on_status — query or push the current state of the contract
  • update / on_update — mutate an existing contract (e.g. reschedule, report payment, add notes)
  • cancel / on_cancel — cancel an order (before or after provider acceptance)
  • rate / on_rate — submit ratings / feedback on provider or farmer
  • support / on_support — request customer support details
Conventions used in the tables below:
  • BAP → BPP means the call originates at the BAP (farmer side) and is routed through ONIX / Beckn Network to the BPP (provider side). The corresponding on_* callback travels in the opposite direction.
  • status calls can be pull (BAP → BPP to query) or push (BPP → BAP as a notification). Both are used.
  • Status values referenced come from LocalServiceFulfillmentAttributes.serviceStatus: ORDER_PLACED, PROVIDER_ACCEPTED, PROVIDER_REJECTED, IN_PROGRESS, COMPLETED, FARMER_CONFIRMED, CANCELLED, CLOSED.

4.6.2 Household Services — Complete Lifecycle

Example: School fee payment, hospital checkup, insurance enrolment, consumer goods delivery.

Structurally identical to Livelihood mapping (4.6.1). The Beckn endpoint sequence and semantics are the same. Category-specific differences are in the payload fields of on_discover / on_select:

  • HouseholdItemAttributes replaces LivelihoodItemAttributes
  • subCategory values: Education, Health, Insurance, Consumer Goods
  • serviceMode enum: SERVICE_DELIVERY / PRODUCT_DELIVERY / FACILITY_VISIT / DIGITAL
  • beneficiaryType (INDIVIDUAL / HOUSEHOLD / CHILD / ELDERLY) and documentsRequired may be populated
  • Some services (e.g., school fee payment) may have scheduledDate as the payment deadline rather than a visit date

All lifecycle phases — discovery, selection, init, confirm, status, update, cancel, rate, support — apply the same way. The full Livelihood table above applies to Household with the above payload differences.

4.6.3 Market Produce — Complete Lifecycle (Reversed Economics)

Example: Farmer sells eggs, milk, farm produce to an aggregator / buyer. Money flows provider → farmer.

The Beckn endpoint sequence is the same as Livelihood. The semantic reversal is in the payload content, not in the protocol flow. Highlights:

Phase Beckn Call Semantic Reversal (vs Livelihood)
Discoverydiscover / on_discover Farmer advertises produce (quantityAvailable, produceCategory); BPP returns aggregators/buyers with buying prices and collectionMode (FARM_GATE_PICKUP / COLLECTION_POINT / FARMER_DELIVERS)
Selectionselect / on_select Quote is the buying price per unit aggregator offers for farmer's produce × quantity
Init / Confirminit, confirm (and on_*) Identical protocol, order created to pick up produce at farmer's location / collection point
Provider acceptancestatus (push) Aggregator accepts the collection request; serviceStatus: PROVIDER_ACCEPTED
Fulfilmentstatus / update Aggregator collects produce, grades it (acceptedGrades), pays farmer; populates providerPaymentReceived to indicate aggregator paid farmer (NOT farmer paid provider). amountReceived = amount paid to farmer.
Farmer confirmationupdate Farmer confirms produce was collected and reports method received (Cash / M-Pesa); farmerPaymentStatus: PAID = farmer confirms they received payment
Cancellationcancel / on_cancel Either side can cancel before collection. After partial collection, use update to adjust quantity actually collected rather than full cancel.
Raterate / on_rate Farmer rates aggregator's fairness (grading, price honoured, payment promptness); aggregator rates produce quality & farmer reliability
Supportsupport / on_support Same contract as Livelihood
