Beckn Protocol Extension Schemas

Domain extension specifications for the NFH-Kuza-Google rural service delivery pilot

Domain Code
local-services
Protocol Version
Beckn v2.0.0
Extension Schemas
7 schemas
Sub-schemas
4 shared types

This document defines the domain extension schemas for a local-services domain within Beckn v2.0.0. The domain enables rural households to discover and transact with local service providers across any service category through a single window, using WhatsApp as the consumer-side BAP interface and a web app as the provider-side BPP interface.

Sub-categories are dynamic. Within each category, sub-categories are managed as master data in the Kuza Platform and are not hardcoded in these schemas. The subCategory field uses a generic {code, name} structure to support any sub-categorization.

Category Codes

Livelihood-Services
Livelihood Flow
Any income/livelihood-related service or input (veterinary, crop, trades/skills services)
Household-Services
Household Flow
Any household need (education, health, insurance, consumer goods)
Market-Produce
Market Flow
Aggregation and sale of any goods/produce. Farmer is the seller, provider is the buyer/aggregator.
Advisory-Content
Advisory Flow
Any informational/training content delivery. Free, discover-only — no order lifecycle.

Livelihood-ServicesLivelihoodYesPaid
Household-ServicesHouseholdYesPaid

Beckn v2 Core Entity Mapping

These extension schemas attach to Beckn v2 core entities through the designated extension containers:

Extension ContainerBeckn v2 Core EntitySchemasPurpose
itemAttributes Resource / Item LivelihoodItemAttributes, HouseholdItemAttributes, MarketProduceItemAttributes, AdvisoryContentItemAttributes Describes what the service/item is and what the consumer needs
offerAttributes Offer LocalServiceOfferAttributes Pricing, payment methods, availability, and capacity
fulfillmentAttributes Performance / Fulfillment LocalServiceFulfillmentAttributes Location, scheduling, delivery mode, and service status tracking
orderAttributes Contract / Order LocalServiceOrderAttributes Payment confirmation, service confirmation, and feedback from both parties
Schema selection is driven by category. A Beckn message for Livelihood-Services would carry LivelihoodItemAttributes in itemAttributes, LocalServiceOfferAttributes in offerAttributes, LocalServiceFulfillmentAttributes in fulfillmentAttributes, and LocalServiceOrderAttributes in orderAttributes. Advisory uses only itemAttributes and fulfillmentAttributes (for location context).

3. MarketProduceItemAttributes

MarketProduceItemAttributes

Container: itemAttributes Applies to: Market-Produce

Describes the aggregation/buying service. In this flow the farmer is the seller and the provider is the buyer/aggregator.

Inverted roles. Unlike other categories where the provider delivers a service to the farmer, in Market-Produce the “item” is the aggregation/buying service. The provider (aggregator) buys produce from the farmer.

7. LocalServiceOrderAttributes

lsor:

LocalServiceOrderAttributes

Container: orderAttributes Applies to: Livelihood-Services, Household-Services, Market-Produce

Post-transaction attributes for all paid flows. Both sides (farmer and provider) report payment status independently, enabling reconciliation without a shared payment gateway.

Dual-sided reporting. Payment is reported independently by both parties — the farmer reports via WhatsApp and the provider reports via the web app. There is no integrated payment gateway; reconciliation happens by comparing both reports.

Service Status Lifecycle

The serviceStatus field in LocalServiceFulfillmentAttributes tracks the order through the following state machine. Advisory content has no lifecycle — it is delivered immediately via contentUrl in on_discover.

State transition rules:

FromToTriggered By
ORDER_PLACEDPROVIDER_ACCEPTEDProvider accepts the order via App
ORDER_PLACEDPROVIDER_REJECTEDProvider rejects the order via App
PROVIDER_ACCEPTEDIN_PROGRESSProvider begins service delivery
IN_PROGRESSCOMPLETEDProvider marks service as done via App
COMPLETEDFARMER_CONFIRMEDFarmer confirms service was received via WhatsApp
FARMER_CONFIRMEDCLOSEDBoth parties have reported; transaction finalized
PROVIDER_REJECTEDCLOSEDAutomatic closure after rejection
Any state before COMPLETEDCANCELLEDEither party cancels
CANCELLEDCLOSEDAutomatic closure after cancellation
Advisory-Content has no lifecycle. Content is delivered immediately via contentUrl in the on_discover response. No status tracking is needed.
