1. Project Overview

NFH-Kuza-Google is a unified service orchestration platform that connects seekers (farmers, patients, households) with providers (agripreneurs, clinicians, facilities, aggregators) across multiple domains through WhatsApp, a lightweight web application, and low-connectivity channels (SMS/USSD, Mobile Lite). The platform is built on the Beckn open network protocol, enabling decentralized discovery and fulfillment of services.

The same underlying flow engine, architecture, technology stack, and data infrastructure serve two live domains today, with a design that admits additional domains without structural change:

  • Local Services / Agri (Beckn domain code: local-services) — Livelihood, Household, Market, and Advisory flows for rural households.
  • Health Services / TIBU (Beckn domain code: health-services) — TB screening, treatment, supervision, and post-TB care for Kenya's National TB Programme.
Design Principle

100% of seeker interaction occurs through channel-agnostic messaging (primarily WhatsApp) — no app installation required. Providers use a lightweight Ionic Angular web app opened via WhatsApp button links, with an optional Mobile Lite interface for field workers (CHVs, CHPs, extension officers) in low-bandwidth conditions.

Service Domains — Agri (local-services)

  • Livelihood Services — agricultural inputs, equipment rental, veterinary services, soil testing, etc.
  • Household Services — plumbing, electrical, cleaning, construction, etc.
  • Market Produce — connecting farmers with aggregators and buyers (reversed economics: buyer pays seeker)
  • Advisory — agricultural content and guidance delivered as media (no order lifecycle)

Service Domains — Health (health-services)

  • Screening & Referral — symptom triage, referral to nearest facility with the right capability (e.g., GeneXpert, X-ray)
  • Treatment Adherence — medication reminders, directly-observed therapy (DOT), supervision by CHP/CHV
  • Lab & Diagnostics — sputum collection, GeneXpert, culture & DST bookings
  • Post-TB Care & Advisory — PTLD monitoring, nutrition, TB-awareness education

1.5 Unified Flow Engine

The central abstraction of the platform is a domain-agnostic 7-step Flow Engine. Every transactional interaction — whether agri or health — is modelled as the same sequence of states. This is what makes the architecture truly unified: verticals differ in schema and vocabulary, but the engine that orchestrates them is identical.

The 7 Steps

Need → Search → Select → Confirm → Deliver → Record → Outcome

  • Need — a seeker has a need (farmer needs vaccination; patient has persistent cough). User trigger; no Beckn call.
  • Search — identify who can help, matching capability + location. Beckn: discover / on_discover.
  • Select — match to best provider, including quote / booking. Beckn: select / on_select + init / on_init.
  • Confirm — both sides agree, binding contract created. Beckn: confirm / on_confirm.
  • Deliver — service rendered (vaccination given; sputum collected). Beckn: status updates (PROVIDER_ACCEPTED, IN_PROGRESS, COMPLETED).
  • Record — data captured (visit notes, clinical record, payment record). Beckn: update + status.
  • Outcome — tracked to result (farmer feedback, treatment success, cure confirmed). Beckn: rate / on_rate plus custom outcome events (e.g., HEALTH_OUTCOME_RECORDED). Can be immediate or longitudinal.

Generalized Actor Model

Specific titles (farmer, patient, agripreneur, clinician) are expressions of four abstract roles. A single individual may play multiple roles across different flows — a CHP is a Provider for screening but a Seeker for supervision.

  • Seeker — any individual or household needing service (farmer, patient, household, CHV, CHP).
  • Provider — anyone delivering service (agripreneur, vet, clinician, lab, agro-vet, training facility).
  • Coordinator — supervisors / oversight (SCTLC, CTLC, extension officer, area manager).
  • Validator — lab, audit, quality assurance.
  • Sponsor — funders, partners, government, NGOs.

Domain & Vertical Hierarchy

Platform (Kuza Flow Engine)
  └── Domain (e.g., local-services, health-services)
        └── Vertical (e.g., poultry, crops, TB, HIV, malaria)
              └── Flow Template (e.g., vaccination-booking, screening-referral)

Each Beckn domain code maps to one platform Domain. Verticals are sub-classifications within a domain. Flow templates are reusable parameterised patterns instantiated by the Flow Engine.

Capability-Based Matching

Providers are matched by service area + structured capability tags, not category alone:

  • Agri: services_offered (vaccination, brooding, AI, soil testing, etc.)
  • Health: capabilities (GeneXpert, X-ray, spirometry, DRTB-certified clinician)

Both stored as structured tags on the Provider profile, queried in discover.

Outcome Recording

Beyond service status COMPLETED, the engine captures an explicit Outcome phase:

  • Agri outcomes: harvest yield (later season), income earned, repeat business, farmer feedback.
  • Health outcomes: cure confirmed, treatment success, prevented mortality, PTLD detected.
  • Outcomes can be captured immediately (feedback at close) or longitudinally (months later), feeding back into analytics workloads for ML training.

3. Architecture Layers

The architecture is organised into six layers plus a set of external integrations. All six layers run on every deployment; external integrations (Beckn Fabric, Google Cloud Services, TIBU, DHIS2) are consumed as needed per domain.

3.1 Layer 1: Frontend Channels

The platform is channel-agnostic. The same flow can be driven over any supported channel; channel choice is a function of the seeker's device and connectivity, not of the domain.

WhatsApp (Seeker-Facing, Primary)

WhatsApp is the primary seeker-facing interface. The bulk of seeker interactions — discovery, provider selection, confirmation, payment declaration, feedback, treatment reminders — occur through WhatsApp messages.

  • Interactive buttons for small binary/ternary choices (Accept/Reject, Yes/No, payment method selection)
  • Numbered text lists for longer option sets (provider lists, service categories, sub-categories)
  • Free text input for location, quantity, symptoms, and other open-ended fields
  • Connected via WhatsApp Cloud API

SMS / USSD (Low-Connectivity Fallback)

For seekers without smartphones or data, the same flow engine is reachable via SMS and USSD shortcodes. Message structure is simplified (numeric menus, short prompts) but the underlying 7-step sequence is preserved.

Mobile Lite (Field Workers)

A TIBUlite-style lightweight mobile client for field workers (CHVs, CHPs, extension officers) operating in low-bandwidth areas. Supports offline queueing of Deliver / Record steps with sync-on-reconnect.

Kuza Web App (Provider-Facing)

A lightweight Ionic Angular web application opened from WhatsApp button links — no native app installation required. Providers (agripreneurs, clinicians, facilities) use it to:

  • Manage their service catalog / capability profile (add/edit services, set pricing, declare capabilities such as GeneXpert, vaccination, etc.)
  • View and accept/reject incoming orders / referrals
  • Update service status (in-progress, completed)
  • Capture clinical / visit records during the Record step
  • Manage their profile and service areas

3.2 Layer 2: Channel Handlers

Design Constraint

The existing webhook.php remains UNTOUCHED. Only routing logic is added to branch marketplace conversations to the new beckn_marketplace_handler.php based on trigger words or message IDs.

  • File: beckn_marketplace_handler.php (new PHP file on Kuza Platform)
  • State machine: manages conversational flow for the 4 marketplace domains (Livelihood, Household, Market, Advisory)
  • Translates WhatsApp interactions into BAP API calls
  • Receives async callbacks from BAP and formats WhatsApp response messages
  • Own tables: conversation state, session tracking — separate from existing poultry chatbot tables (chatbot_nodes, chatbot_transitions)
  • Provider notifications are triggered via a messaging queue/cron job, NOT through webhook.php

3.3 Layer 3: Flow Engine

The Flow Engine sits above BAP/BPP and below the Channel Handlers. It is the domain-agnostic state machine that implements the 7-step lifecycle (Need → Search → Select → Confirm → Deliver → Record → Outcome) and decides which Beckn call (if any) each step translates into.

  • Flow templates per domain / vertical (vaccination-booking, screening-referral, market-sell-produce, treatment-adherence) are loaded at runtime.
  • Transaction state is persisted per seeker session (flow_instance_id) and correlated with the Beckn transaction id.
  • Routing — each step may produce a BAP call, a BPP-originated event, or a pure UX transition (e.g., the Need step has no network call).
  • Progressive disclosure — the same template can render differently by actor role (seeker vs. coordinator vs. validator).
  • Optional adoption — the engine is not required for read-only analytics; existing non-flow interactions (one-off surveys, content pulls) continue to operate independently.

3.4 Layer 4: Beckn Layer (BAP + BPP)

The Beckn layer hosts both the seeker-side participant (BAP) and provider-side participant (BPP), each paired with its ONIX adapter. Both are Kuza-built, hosted on GCP, and communicate with the external Beckn Fabric (see External Integrations, 3.7).

BAP (Beckn Application Platform) — Seeker-side

  • Receives translated requests from the Flow Engine
  • Constructs Beckn v2 protocol messages: discover, select, init, confirm, status, cancel, rate
  • Sends protocol messages through the ONIX-BAP adapter (Go / Docker)
  • Receives async callbacks (on_discover, on_select, on_init, on_confirm, on_status, on_cancel, on_rate)
  • Pushes results back to the Flow Engine for seeker delivery

BPP (Beckn Provider Platform) — Provider-side

  • Receives Beckn protocol requests from the network via ONIX-BPP adapter (Go / Docker)
  • Connects to the Kuza Platform to fetch provider catalogs, service areas, and pricing
  • Manages order lifecycle on the provider side (create, accept, update status, complete)
  • Sends async Beckn responses back through ONIX-BPP
  • Serves the Kuza Web App (provider interface) for catalog and order management

ONIX Adapters

  • Plugin-based Go servers running the fidedocker/onix-adapter Docker image
  • Handle message signing, signature validation, schema validation, and network routing against Beckn Fabric
  • Hosted on GCP as Docker containers on the same instance as BAP/BPP application logic

3.5 Layer 5: Data Foundation (Kuza Platform)

MySQL 8 (with a PostgreSQL migration under way in kuza-next) on GCP serves as the single source of truth. The Beckn network is stateless — all persistent data lives here.

Sovereign Data & Compliance

Seeker / patient / farmer data stays in country. Records are persisted with a full audit trail (status transitions, who updated, when). For the health domain, the record schema is designed to be FHIR-compatible so clinical records can be exported into national health information systems without reshaping.

Existing Tables Used

  • user — unified seeker / provider identity (agripreneurs, farmers, clinicians, patients, CHPs)
  • Partner and organization data
  • Content library (for Advisory domain, agri & health)

3.6 Layer 6: Cloud Infrastructure (GCP)

The entire Kuza-built stack is deployed on Google Cloud Platform (GCP). This is the foundational hosting layer on which Layers 1 through 5 run.

  • Compute: hosts the Channel Handlers, Flow Engine, BAP, BPP, Kuza Web App, ONIX adapters (Docker)
  • Database: hosts the Kuza Platform data store (MySQL 8 / PostgreSQL)
  • Networking: public endpoints, TLS termination, load balancing for BAP / BPP callbacks
  • Co-location benefit: Kuza application code and Google Content APIs / AI interfaces are co-located on the same cloud, minimising latency for in-flow API calls
  • Sovereign data: patient / seeker / farmer data stays in-country (Kenya region) per DPDP-style compliance requirements

3.7 External Integrations

The platform's six architectural layers talk to several external services. These are not layers of the architecture; they are third-party systems the platform integrates with.

3.7.1 Beckn Fabric (external Beckn Network services)

  • DeDi Registry — identity & routing for BAP / BPP participants (fabric.nfh.global/registry/dedi)
  • Catalog Service — BPP publishes catalogs (fabric.nfh.global/beckn/catalog)
  • Discovery Service — handles discover requests, queries Catalog Service
  • All hosted externally by Beckn Fabric; Kuza connects via ONIX adapters

3.7.2 Google Content APIs — Curated Domain Content

Google publishes curated content APIs per domain (agri, health, market, finance, etc.) that serve informational / educational content via the chatbot. These are separate from Beckn transactional catalog data — they don't list providers for sale, they deliver knowledge.

  • Primary consumer: Advisory flow — BPP calls Content APIs during on_discover to source advisory material (articles, videos, audio, infographics, structured guides)
  • Secondary consumers: Livelihood / Household / Market flows — optional enrichment (e.g., "Best practices for vaccinating broilers" attached to a Vaccination order)
  • Called from: BPP application logic (during request processing); optionally from the Channel Handler for inline content

3.7.3 Google AI Interfaces — Intelligent Processing

Distinct from content APIs, the AI interfaces provide computational services that process input and return derived results.

  • Location normalization (all flows): free-text location → structured County + Sub-County
  • Advisory content tailoring (Advisory flow): combines Content API output with seeker context to personalise / summarise / translate
  • Future: intent extraction from free-text queries, language translation, sentiment analysis on feedback
  • Called from: Channel Handler (location normalization) and BPP (advisory tailoring)

3.7.4 Health Domain Integrations (for health-services)

  • TIBU Registers — Kenya's National TB Programme data spine (1.3M+ patient records, 23 linked registers). UUID-linked; read / write via REST.
  • DHIS2 — aggregated indicator sync with national / county health systems
  • FHIR R4 endpoints — optional standard for health data exchange
Analytics is a logical capability, not a layer. Every flow emits event / outcome data into the Kuza Platform (Layer 5). Analytics and ML run as independent workloads consuming this data. See the specification document's Phased Analytics Roadmap appendix for how analytics maturity progresses in phases (descriptive → predictive → surveillance).

4.6 Complete Beckn Endpoint Mapping

This section maps every user-facing step in each flow to the corresponding Beckn v2 protocol endpoint, covering the full order lifecycle — discovery, selection, initialization, confirmation, fulfilment status, in-flight updates, cancellation, rating/feedback, and support.

Beckn endpoints reference:

  • discover / on_discover — find services/providers in the network
  • select / on_select — request a priced quote for a specific item
  • init / on_init — submit billing & fulfilment details, receive draft order
  • confirm / on_confirm — create the binding order
  • status / on_status — query or push the current state of the contract
  • update / on_update — mutate an existing contract (e.g. reschedule, report payment, add notes)
  • cancel / on_cancel — cancel an order (before or after provider acceptance)
  • rate / on_rate — submit ratings / feedback on provider or farmer
  • support / on_support — request customer support details
Conventions used in the tables below:
  • BAP → BPP means the call originates at the BAP (farmer side) and is routed through ONIX / Beckn Network to the BPP (provider side). The corresponding on_* callback travels in the opposite direction.
  • status calls can be pull (BAP → BPP to query) or push (BPP → BAP as a notification). Both are used.
  • Status values referenced come from ServiceFulfillmentAttributes.serviceStatus: NEED_REGISTERED, SEARCHED, SELECTED, CONFIRMED, DELIVERED, RECORDED, OUTCOME_RECORDED, CANCELLED, REJECTED, LOST_TO_FOLLOWUP, CLOSED.

4.6.2 Household Services — Complete Lifecycle

Example: School fee payment, hospital checkup, insurance enrolment, consumer goods delivery.

Structurally identical to Livelihood mapping (4.6.1). The Beckn endpoint sequence and semantics are the same. Category-specific differences are in the payload fields of on_discover / on_select:

  • Same ServiceItemAttributes schema — verticalAttributes.vertical changes from "agri-livelihood-poultry" to "household-education" / "household-health" etc.
  • subCategory values: Education, Health, Insurance, Consumer Goods
  • serviceMode enum: SERVICE_DELIVERY / PRODUCT_DELIVERY / FACILITY_VISIT / DIGITAL
  • beneficiaryType (INDIVIDUAL / HOUSEHOLD / CHILD / ELDERLY) and documentsRequired may be populated
  • Some services (e.g., school fee payment) may have scheduledDate as the payment deadline rather than a visit date

All lifecycle phases — discovery, selection, init, confirm, status, update, cancel, rate, support — apply the same way. The full Livelihood table above applies to Household with the above payload differences.

7. Deployment Topology

All Kuza-built components are deployed on Google Cloud Platform (GCP), in-country to preserve data sovereignty for patient and farmer records. Beckn Fabric services are externally hosted and accessed over HTTPS.

Health-domain deployment note

For health-services deployments, the Kuza Platform data store exposes a FHIR-compatible export adapter so clinical records can be synchronised with MoH systems without reshaping. Audit trail (who changed what, when) is retained on every record, and seeker / patient PII never leaves the in-country GCP region.

9. Architectural Principles

Seven principles guide the design and evolution of the platform. They originate in TIBU (the health deployment) and have been generalised to apply to all domains:

  1. Preserve, don't replace — existing data systems (Kuza MySQL, TIBU registers, MoH systems) stay; the platform adds orchestration on top. No rip-and-replace.
  2. UUID is the spine — every entity (seeker, provider, order, flow instance, outcome) has a universal UUID so records can be linked across systems, domains, and time.
  3. Channel-agnostic intelligence — the same decision / flow step can be delivered via WhatsApp, Web, SMS/USSD, or Mobile Lite. Channel is a rendering concern, not a logic concern.
  4. Domain-agnostic infrastructure — the same Flow Engine handles agri and health. Adding a domain is a configuration of templates, schemas, and taxonomies — not new infrastructure.
  5. Sovereign data — patient / farmer data stays in country, with full audit trail. Schemas are compliant (FHIR-compatible for health; auditable for agri).
  6. Progressive disclosure — UI complexity matches user role. A seeker sees a short prompt; a coordinator sees a dashboard; a validator sees an audit view. Same underlying data, different surfaces.
  7. Flow Engine is optional until proven — descriptive analytics can run on existing data without the engine. The engine earns its place as outcome data accumulates and predictive capabilities become viable.
