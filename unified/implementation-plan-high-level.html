NFH-Kuza-Google Pilot — High-Level Implementation Plan

Version: 1.0
Date: April 2026
Status: Draft for review

Introduction

This document sets out the high-level implementation plan for the NFH-Kuza-Google pilot. The pilot delivers a unified, domain-agnostic service-orchestration platform for rural households in Kenya. It spans two Beckn domains — local-services (agri / rural) and health-services — sharing a single Flow Engine, channel layer, data foundation, and integrations with Google Cloud services and the Beckn network.

The plan is organised into eight phases. Each phase has:

  • a single Goal statement,
  • a list of Inputs Required — named operational artifacts (lists, datasets, content, credentials, decisions) that must be provided before the phase can start, with the provider of each artifact identified, and
  • a set of Expected Outputs — concrete deliverables produced by the phase.

The intent of this document is to surface every input that the pilot team needs from partners and from internal teams. Once the high-level plan is reviewed and approved, a detailed plan per phase will follow, breaking each phase into tasks, owners, and durations.

Phase Summary

# Phase One-line Goal
0 Project Mobilisation Governance + environments
1 Foundation — Data Model & Master Data Schema, seeds, data dictionary
2 Beckn Network Onboarding ONIX running, BAP/BPP registered
3 Core Platform Services Flow Engine + BAP + BPP running
4 Channel Integration WhatsApp Flow Handlers, Web App, Google AI live
5 Vertical Rollout — Local Services 4 agri/rural flows live
6 Vertical Rollout — Health Services 6 health flows live, cascades + batches
7 Pilot Launch Real cohort on the system
8 Stabilisation & Analytics KPIs, dashboards, feedback loop

Sequencing notes:

  • Phase 0 unblocks everything else.
  • Phases 1 and 2 can run in parallel once Phase 0 is complete.
  • Phase 3 depends on Phases 1 and 2.
  • Phase 4 depends on Phase 3.
  • Phases 5 and 6 default to sequential (Local Services first, then Health Services), though they can parallelise if two independent squads are available.
  • Phase 7 is gated by Phase 5 and Phase 6 (whichever subset is in scope for launch).
  • Phase 8 begins with pilot go-live and runs continuously thereafter.

Critical-Path Items

Inputs that most often delay pilots of this shape. Flagging them here so owners can start these earlier than the phase in which they are strictly required:

  1. Master-data lists (Phase 1) — geography is usually fine; vertical taxonomies and sub-categories can stall waiting on SMEs. Start SME conversations during Phase 0.
  2. WhatsApp template approvals (Phase 4) — Meta’s approval can take days per template; batch-submit templates early, even if Phase 4 has not formally begun.
  3. Facility capability + CHP roster data (Phase 6) — these usually live in spreadsheets across partner organisations and need reconciliation. Kick off data-cleanup with the health partner during Phases 1–2.
  4. Predefined feedback-option lists per vertical (Phase 5) — easy to defer, hard to retrofit. Capture these during Phase 1 master-data work if possible.
  5. Cascade trigger rules (Phase 6) — require clinical sign-off and often get stuck in review. Start clinical review conversations during Phase 3 even though the rules are consumed in Phase 6.
