NFH-Kuza-Google — Beckn Schema Definitions (Multi-Domain)

Extension schemas for local-services (Agri/Rural) AND health-services (TIBU/Health) domains

Beckn Domains
local-services · health-services
Protocol Version
Beckn v2.0.0
Extension Schemas
14 schemas (7 + 7)
Sub-schemas
4 shared · 2 health-specific

This document defines the Beckn v2.0.0 extension schemas used by the NFH-Kuza-Google platform across two Beckn domains:

  • local-services — the existing agri/rural domain covering livelihood services, household services, market produce aggregation, and advisory content.
  • health-services — the new health domain (TIBU integration) covering TB screening, TB treatment, supervision visits, and post-TB care / health advisory.

Both domains run on a shared flow-engine architecture. Common patterns — the generic subCategory: {code, name} structure, capability matching, and the unified service status lifecycle — appear in both, differing only in the domain-specific values and sub-schemas. Shared sub-schemas (ServiceArea, QuantitySpec, Feedback, and the generic seeker location) are reused across both domains.

Sub-categories are dynamic. Within each category, sub-categories are managed as master data in the Kuza Platform and are not hardcoded in these schemas. The subCategory field uses a generic {code, name} structure to support any sub-categorization.
Choosing a schema set. See the Domain Selection Guide below to decide which Beckn domain and itemAttributes schema applies to your use case.

Beckn v2 Core Entity Mapping

Both domains attach their extension schemas to the same set of Beckn v2 containers. The container is identical; only the extension schema plugged into it changes per domain.

Extension ContainerBeckn v2 Core Entitylocal-services Schemashealth-services Schemas
itemAttributes Resource / Item LivelihoodItemAttributes, HouseholdItemAttributes, MarketProduceItemAttributes, AdvisoryContentItemAttributes HealthScreeningItemAttributes, HealthTreatmentItemAttributes, HealthSupervisionItemAttributes, HealthAdvisoryItemAttributes
offerAttributes Offer LocalServiceOfferAttributes HealthServiceOfferAttributes
fulfillmentAttributes Performance / Fulfillment LocalServiceFulfillmentAttributes HealthServiceFulfillmentAttributes
orderAttributes Contract / Order LocalServiceOrderAttributes HealthServiceOrderAttributes
Schema selection is driven by domain and category. Both domains use the same four Beckn containers — just with domain-specific extensions plugged in. A local-services message for Livelihood-Services carries LivelihoodItemAttributes; a health-services message for TB-Screening carries HealthScreeningItemAttributes. Advisory categories in either domain may use only itemAttributes and fulfillmentAttributes.

Container: itemAttributes Applies to: Market-Produce

Describes the aggregation/buying service. In this flow the farmer is the seller and the provider is the buyer/aggregator.

The serviceStatus field in the fulfillment extension of either domain tracks the order through a unified 7-stage flow-engine lifecycle. health-services uses the full enum including the OUTCOME_* and LOST_TO_FOLLOWUP states. local-services uses a compatible subset (legacy names are retained for the existing agri flows and map onto the new states as shown below). Advisory content in either domain has no lifecycle — it is delivered immediately in on_discover.
