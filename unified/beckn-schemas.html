1. Overview

The Unified 4-Schema Model

This document defines four generic, domain-agnostic extension schemas that plug into the Beckn v2 protocol. Previously the protocol exposed 14 domain-specific schemas (7 for agri/rural, 7 for health); those have been collapsed into this single set of four:

SchemaPrefixContainerPurpose
ServiceItemAttributessia:itemAttributesDescribes the service / product / content / advisory on offer or being sought
ServiceOfferAttributessoa:offerAttributesPricing, availability, payment methods, cancellation policy
ServiceFulfillmentAttributessfa:fulfillmentAttributesLocation, scheduling, delivery mode, lifecycle status
ServiceOrderAttributessor:orderAttributesPayment, feedback, outcome capture, consent & audit

The verticalAttributes Pattern

Every generic schema exposes an optional open-object field called verticalAttributes. This is a free-form key-value bag where a specific vertical (poultry vaccination, TB screening, school fees, etc.) can carry its domain-specific fields without requiring a schema change.

By convention, every verticalAttributes object MUST include:

  • vertical Required — the vertical identifier, e.g. "agri-livelihood-poultry", "health-tb-screening", "household-education".

The OpenAPI schema validates that verticalAttributes is an object. Field-level validation happens at the application layer, driven by the vertical value. The “Vertical Cookbook” (Section 6) documents the conventional shape for each known vertical.

Adding a new use case no longer requires a schema change. To onboard a new vertical, simply document what verticalAttributes should contain. The four generic schemas stay stable; the Beckn protocol stays stable; only the application layer learns a new vertical-to-field mapping.

2. Beckn Domain Codes & Categories

The same four generic schemas are used for both Beckn domain codes. The vertical discriminator inside verticalAttributes tells the application layer which vertical-specific contract applies.

3. Generic Schemas

Each of the four schemas below is domain-agnostic. Any vertical across any Beckn domain plugs into them; vertical-specific data lives inside the verticalAttributes bag.

5. Service Status Lifecycle

The serviceStatus field in ServiceFulfillmentAttributes follows a unified 8-stage lifecycle that applies universally, regardless of Beckn domain or vertical. Advisory content is the only exception — it has no lifecycle and is delivered directly in on_discover.

State transition rules:

FromToTriggered By
NEED_REGISTEREDSEARCHEDBAP issues discover; catalog returned
SEARCHEDSELECTEDSeeker selects a provider/item
SELECTEDCONFIRMEDProvider accepts the order
CONFIRMEDDELIVEREDProvider delivers the service
DELIVEREDRECORDEDProvider records completion (notes, data)
RECORDEDOUTCOME_RECORDEDOutcome captured (seeker confirmation OR clinical result OR sale completion)
DELIVERED / RECORDEDLOST_TO_FOLLOWUPSeeker cannot be contacted
OUTCOME_RECORDED / LOST_TO_FOLLOWUPCLOSEDTransaction finalized
Any stage before DELIVEREDCANCELLEDEither party cancels
After DELIVEREDREJECTEDSeeker disputes delivered service
CANCELLED / REJECTEDCLOSEDAutomatic terminal closure
Advisory content has no lifecycle. Any vertical whose subCategory.code is advisory-content (or equivalent) is delivered immediately via contentUrl in the on_discover response.

7. Beckn Core Entity Mapping

The four generic extension containers attach to the same set of Beckn v2 core entities regardless of domain or vertical. Only the contents differ.

Extension ContainerBeckn v2 Core EntityWhat it carries
itemAttributes Resource / Item ServiceItemAttributes — what is being offered or sought
offerAttributes Offer ServiceOfferAttributes — pricing, coverage, availability, cancellation
fulfillmentAttributes Performance / Fulfillment ServiceFulfillmentAttributes — location, scheduling, delivery, lifecycle state
orderAttributes Contract / Order ServiceOrderAttributes — payment, feedback, outcome, consent, adverse events
Stable Beckn surface. Onboarding new verticals never requires a new Beckn container or a new extension schema. The four generic schemas remain stable; the application layer reads verticalAttributes.vertical to dispatch vertical-specific logic.

8. Adding a New Vertical

Suppose you want to add an advisory-finance vertical (financial literacy content) under the local-services domain. No schema change is required. The steps are:

  1. Pick the Beckn domain. Here: local-services (existing rural domain).
  2. Pick the category code. Here: Advisory-Content.
  3. Pick a vertical identifier. Convention: <area>-<sub-area>. Here: advisory-finance.
  4. Document the verticalAttributes shape. Publish it as an extension of this cookbook: 
    {
  "vertical": "advisory-finance",
  "topic": "Personal Finance",
  "subTopic": "Savings Groups",
  "sourceProvider": "Kuza Finance Academy"
}
  5. Wire the application layer. When the application sees verticalAttributes.vertical === "advisory-finance", it routes to the finance-vertical handler. No Beckn protocol change, no OpenAPI change.
No schema change = no breaking change. Existing integrations keep working. Only consumers that want to read / write the new vertical's fields need updates.
